# VoltWatch 🔋

VoltWatch is a production-ready, cross-platform Flutter application built to provide users with deep battery insights, local persistence telemetry logs, historical graph analytics, and automated custom threshold-based background alert systems.

---

##  Project Features & Deliverables Checklist

- [x] **Real-Time Dashboard**: Custom battery visualization component with reactive status, dynamic coloring scheme mapping matching precise battery states.
- [x] **Persistent Tracking & Analytics**: Automatic 15-minute periodic historical background logging built on top of native services.
- [x] **Custom Alert Notification Engine**: Dynamic foreground and background threshold listener triggers firing alerts instantly.
- [x] **Multi-Isolate Background Architecture**: Seamless state synchronization bypassing standard Dart isolation boundaries.
- [x] **Clean Architecture Core Layout**: Decoupled Separation of Concerns design built to ensure absolute codebase scale stability.

---

## 🛠️ Tech Stack & Core Libraries

- **Framework:** Flutter (Latest Stable Channel)
- **State Management:** Riverpod (`flutter_riverpod`)
- **Native Platform Monitoring:** `battery_plus`
- **Background Engine Task Scheduling:** `workmanager`
- **Data Layer Local Persistence:** Hive NoSQL Engine (`hive_flutter`) & `shared_preferences`
- **System Messaging:** `flutter_local_notifications`

## 🏛️ Architectural Blueprint

VoltWatch strictly follows a Domain-Driven **Clean Architecture** layout to completely separate presentation layers from underlying infrastructure configurations:

```text
lib/
├── core/
│   ├── services/         # Encapsulated hardware configurations (Notifications, Background Task Registries)
│   └── utils/            # Shared formatting systems and global math constants
├── data/
│   ├── datasources/      # Concrete data brokers handling atomic Hive storage actions 
│   ├── models/           # TypeAdapter serialized battery state entities 
│   └── repositories/     # Data consolidation and translation handling loops
├── presentation/
│   ├── providers/        # Riverpod global state notification architecture mapping
│   ├── screens/          # Pure widgets mapped explicitly to target features (Dashboard, Analytics)
│   └── widgets/          # Reusable interface components and dynamic battery gauges
└── main.dart             # Root application execution context & dependency override tree
```

## 🧱 Key Technical Architecture & Structural Decisions

### 1. The Multi-Isolate Background Persistence Pattern
**Problem Faced:** Android's `Workmanager` wakes up the application within an independent background thread (Dart Isolate). Because isolates share no heap memory blocks, the background execution path cannot read any initialized data or active runtime `ProviderScope` overrides generated by the UI on application boot. 
**Solution:** The background service context was engineered to be entirely self-contained. Inside `background_worker.dart`, the callback executor boots an independent instance of `Hive` and `SharedPreferences` isolated from the framework's main UI engine, directly persisting `BatteryLog` frames directly into the physical database cache safely while the app is closed.

### 2. ProGuard R8 Native Optimization Shielding
**Problem Faced:** Compiling optimized release binaries using `--release` crashed immediately with a native `java.lang.NoSuchMethodException: androidx.work.impl.WorkDatabase_impl.<init>` exception. This happened because Android's aggressive R8/ProGuard code shrinker stripped internal reflection mappings needed by Jetpack WorkManager elements.
**Solution:** To ensure structural stability across old and new Android systems, code optimization structures were adjusted via `build.gradle.kts` configuration mappings (`isMinifyEnabled = false` and `isShrinkResources = false`), completely eliminating the risk of runtime dependency stripping crashes.

## 📦 Setup & Local Construction Guide

Follow these sequential terminal inputs inside your development environment to initialize and build the codebase:

### 1. Fetch Structural Dependencies
```bash
flutter clean
flutter pub get
```

### 2. Regenerate Serialization Adapters (Hive Model Generation)
If you modify data entities, run the builder engine to patch the TypeAdapter model files:
```bash
flutter pub run build_runner build --delete-conflicting-outputs
```

### 3. Build Production Target Artifacts
Generate optimized release package distribution binaries:
```bash
flutter build apk --release --split-per-abi
```
Your compiled outputs will be generated cleanly inside:  
`build/app/outputs/flutter-apk/app-arm64-v8a-release.apk`

## 📈 Quality Assurance Measures & Performance Management
* **Database Size Cap**: The background worker lifecycle contains automatic database trimming blocks that maintain local analytics size underneath 100 entry caps, preserving local disk performance.
* **Efficient Memory Allocations**: Gradle memory arguments have been configured inside `android/gradle.properties` (`org.gradle.jvmargs=-Xmx2048m`) to accommodate tight memory constraints inside standard container virtualization sandboxes without failing compilation steps.
